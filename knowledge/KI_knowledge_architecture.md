# KI: Architectural Decisions (ADR)

## Overview
Процесс документирования значимых архитектурных изменений, выбора технологий и логических переходов в проекте.

## Key Components

| Component | File | Purpose |
|---|---|---|
| `/create-adr` | `.know/workflows/create-adr.md` | Вворкфлоу для создания новых записей решений. |
| **ADR Repository** | `.know/decisions/` | Хранилище всех принятых решений в хронологическом порядке. |
| **ADR Template** | `.know/decisions/000_adr_template.md` | Стандартный шаблон для фиксации контекста, выбора и последствий. |

## Decision Process
- **Chronology**: Каждое решение получает инкрементальный ID (`001`, `002`...).
- **Immutable State**: Старые ADR никогда не редактируются. Если решение пересматривается, создается новый ADR со ссылкой на предыдущий.
- **Append-Only Log**: Внутренняя структура ADR подразумевает последовательное накопление контекста и уроков.

## ADR Structure
1. **Context and Problem**: Почему старый подход перестал работать?
2. **Decisions & Lessons**: Что пробовали и к чему пришли в итоге.
3. **Impact**: Как это изменит работу над проектом и какие есть компромиссы.

## Common Pitfalls
- **Skipping Registration**: Создание ADR без добавления его в `doc_config.json` приведет к тому, что другие агенты не увидят это решение в своем контексте.
- **Vague Impact**: Описание влияния без конкретных компромиссов (trade-offs) делает ADR бесполезным для будущего анализа.
