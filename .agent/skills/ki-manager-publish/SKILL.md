# ki-manager: Публикация новой версии

## Overview

Скилл выполняет полный цикл релиза пакета `ki-manager` на PyPI через GitHub Actions:
- обновляет версию в двух файлах,
- создаёт git-коммит и тег,
- пушит тег на origin (GitHub Actions публикует пакет автоматически).

## Trigger phrases

Запускается при любом из этих запросов (или схожих по смыслу):
- «запушь новую версию»
- «создай версию пакета»
- «опубликуй релиз»
- «сделай релиз»

Если пользователь просит «сделай коммит» / «запушь изменения» **без упоминания версии** — ОБЯЗАТЕЛЬНО спроси:
> «Хочешь попутно выпустить новую версию пакета на PyPI или просто сделать коммит?»

## Workflow

### Шаг 0: Уточнение версии

Перед тем как что-либо менять:

1. Прочитай текущую версию из `pyproject.toml` (строка `version = "..."`).
2. Предложи следующую patch-версию (например `2.0.11` → `2.0.12`).
3. Спроси пользователя, подтверждает ли он эту версию или хочет указать другую.

> [!IMPORTANT]
> Не меняй версию без явного подтверждения пользователя.

### Шаг 1: Проверить состояние репозитория

```powershell
git status
git log -n 3 --oneline
```

Убедись, что рабочая директория чистая или все нужные изменения добавлены. Если есть незакоммиченные файлы — спроси пользователя, добавлять ли их в этот коммит.

### Шаг 2: Обновить версию в двух файлах

#### `pyproject.toml`
```toml
[project]
version = "X.Y.Z"  # ← обновить
```

#### `src/ki_manager/__init__.py`
```python
__version__ = "X.Y.Z"  # ← обновить синхронно
```

> [!CAUTION]
> Версии в обоих файлах ДОЛЖНЫ совпадать. Если они разойдутся — PyPI примет пакет со старой версией.

### Шаг 3: Добавить файлы и создать коммит

```powershell
git add pyproject.toml src/ki_manager/__init__.py
# Если пользователь добавил другие файлы — включить их тоже
git commit -m "Bump version to X.Y.Z"
git push origin main
```

Сообщение коммита по умолчанию: `"Bump version to X.Y.Z"`. Спроси пользователя, если он хочет другое сообщение.

### Шаг 4: Создать и запушить тег

```powershell
git tag vX.Y.Z
git push origin vX.Y.Z
```

> [!CAUTION]
> Тег ДОЛЖЕН быть создан на том же коммите, в котором присутствует файл `.github/workflows/publish.yml`. Никогда не создавай тег на старом коммите.

### Шаг 5: Подтвердить успех

После выполнения всех команд сообщи пользователю:
- Какая версия опубликована
- Что GitHub Actions запустится автоматически (можно проверить на вкладке Actions на GitHub)
- После успеха пакет появится на PyPI
- Если пользователь установил `ki-manager` через `uvx` — потребуется один раз запустить `uvx --refresh ki-manager` для обновления кэша

## Common Mistakes

1. **Версии в `pyproject.toml` и `__init__.py` не совпадают** — PyPI отклонит пакет или загрузит неправильную версию.
2. **Тег создан на старом коммите** — `publish.yml` не окажется в истории этого коммита и GitHub Actions не запустится.
3. **`uvx` кэш не обновлён** — после публикации пользователям нужно запустить `uvx --refresh ki-manager` хотя бы один раз.
